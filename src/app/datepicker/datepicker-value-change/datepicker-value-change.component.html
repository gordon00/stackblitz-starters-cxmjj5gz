L'exemple suivant sert à démontrer que si on utilise une ValidatorFn sur le FormControl, elle finit par être bypassée par le Validator du datepicker.

On applique donc un validator qui renvoit tout le temps l'erreur 'alwaysInError' dans le cadre de cet exemple

  • Dans le champ ci-dessous, saisir une date partielle 01/01 (ou complète). On constate pendant la saisie la présence de l'erreur 'alwaysInError'
  • Sortir du champ (blur)
  • Le FormControl reste bien invalide (OK)
  • Mais les indicateurs visuels d'erreur disparaissent (KO)

Résultat attendu : Une ValidatorFn ajoutée dans le FormControl utilisé par le datepicker ne devrait pas être invisibilisés.

Analyse : il n'est pas possible d'utiliser les @Input 'severity' et 'message' car les valeurs sont surchargées par la méthode DsfrDatepickerComponent.handleInternalValidation().

Demande d'évolution : Dans le cadre de l'utilisation du composant datepicker avec les reactive forms, il faudrait pouvoir désactiver le rendu interne des messages d'erreur. Nous souhaiterions avoir la main sur la transformation des erreurs en message en utilisant les @Input message et severity

Nous avons nos propres mécanismes de transformation des erreurs de FormControl en message d'erreur. Aussi, il serait génial que les erreurs (cf. DsfrDatepickerComponent.validate()) 'err_min_date' et 'err_max_date' contiennent respectivement la valeur minDate et maxDate afin de pouvoir gérer nous même le contenu des messages d'erreur internes en y incluant des paramètres.
Exemple : Le date doit être postérieure au 01/05/2025.

Mais il est important pour nous de garder un contrôle total du message car celui-ci pourrait être différent selon le contexte. Pour le même type d'erreur mais dans un écran différent, on pourrait également vouloir le message : "La date doit être postérieure à la date du jour."
